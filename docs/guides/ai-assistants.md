# AI Assistants

typesugar is a compile-time macro system, which means an AI coding assistant
that doesn't know that will confidently break your build — by "cleaning up" a
macro import it thinks is unused, by hand-writing the code a `@derive` would
generate, or by removing the bundler plugin to "simplify" the config.

typesugar ships the context to prevent that, and `typesugar init` writes it
into your project.

## What gets scaffolded

```bash
npx typesugar init          # asks (default: yes)
npx typesugar init --ai     # write it, no prompt, and install the Claude Code skill
npx typesugar init --no-ai  # skip entirely
```

| File                        | Purpose                                                                      |
| --------------------------- | ---------------------------------------------------------------------------- |
| `AGENTS.md`                 | The context itself — read natively by Cursor, Copilot, Codex, Zed and others |
| `CLAUDE.md`                 | A one-line pointer at `AGENTS.md` (only created if you don't have one)       |
| `.claude/skills/typesugar/` | A Claude Code skill that fires on typesugar errors and macro debugging       |

`typesugar create` scaffolds all of it into new projects automatically.

### AGENTS.md, not five per-tool files

`AGENTS.md` is the emerging cross-tool convention, so one file covers Cursor,
Copilot, Codex, Zed and friends. typesugar deliberately does **not** write
`.cursor/rules/*.mdc` or `.github/copilot-instructions.md` — duplicating the
same content per tool is drift surface, not coverage.

Claude Code reads `CLAUDE.md`, so it gets a pointer file — and only if you
don't already have one. An existing `CLAUDE.md` is yours; typesugar will never
edit it, it just prints the line to add:

```
See @AGENTS.md for typesugar usage rules.
```

## It's safe to re-run

The scaffolded content lives between markers:

```markdown
<!-- typesugar:begin (generated by typesugar init — re-run to update) -->

...

<!-- typesugar:end -->
```

Re-running `typesugar init` (say, after upgrading typesugar) refreshes **only**
that block. Anything you wrote around it — your own conventions, other tools'
instructions — is preserved exactly.

## What the context tells an assistant

- **Setup invariants:** keep the bundler plugin and the tsconfig `plugins`
  entry; use `typesugar check` rather than bare `tsc --noEmit` (which sees
  pre-expansion code and reports artifacts that aren't real errors); never
  delete a macro import that looks unused.
- **The macro forms:** `comptime`, `@derive` (JSDoc _and_ decorator spellings),
  `@impl`/`summon`, companion access (`Point.Eq.equals(a, b)`), and that
  `match` is expression-based (`.then(value)`, not `.then(callback)`).
- **The debugging loop:** `typesugar doctor` → `typesugar expand --diff` →
  `typesugar check`, and that the expansion is ground truth — check it before
  theorizing about a runtime bug.
- **Where to read more:** the [error catalog](/errors/) (one page per
  diagnostic) and the machine-readable docs below.

## Machine-readable docs

The whole documentation site is published in the [llms.txt](https://llmstxt.org)
format, so an assistant can pull exactly what it needs:

| File                                                    | What it is                                      |
| ------------------------------------------------------- | ----------------------------------------------- |
| [`/llms.txt`](https://typesugar.org/llms.txt)           | Index: one line per page, with descriptions     |
| [`/llms-full.txt`](https://typesugar.org/llms-full.txt) | The entire corpus in a single file              |
| [`/errors/`](/errors/)                                  | One page per diagnostic code, with worked fixes |

Point your assistant at these when it needs detail beyond the scaffolded
context — for example: _"read https://typesugar.org/llms-full.txt and then add
a `@derive(Ord)` instance to this type"_.
