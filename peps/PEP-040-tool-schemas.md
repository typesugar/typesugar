# PEP-040: `@tool` — Compile-Time Agent Tool Schemas

**Status:** Draft
**Date:** 2026-06-10
**Author:** Dean Povey

## Context

Every team building LLM agents writes tool definitions: a JSON Schema describing
the tool's parameters, a description for the model, and runtime validation of the
arguments the model sends back. Today this is done one of three ways:

1. **Hand-written JSON Schema** — drifts from the TypeScript signature immediately.
2. **Zod (or similar) + `zod-to-json-schema`** — the schema is duplicated as a
   runtime value, adds a dependency, and the TypeScript type is *derived from* the
   schema rather than being the source of truth.
3. **Decorator + reflect-metadata frameworks** — require `emitDecoratorMetadata`,
   lose precision (no unions, no literals), and carry runtime weight.

typesugar already has every primitive this problem needs: `typeInfo<T>()` reads
full structural type information from the checker, `validator<T>()` generates
zero-dependency validation code, and JSDoc comments are available on declarations
at transform time. This PEP composes them into a single wedge feature.

**Strategic framing:** the ROADMAP says typesugar's problem is adoption, not
features. `@tool` is deliberately chosen as a wedge: it is small, self-contained,
solves a problem every AI team has *this year*, requires only the unplugin (no
typeclasses, no operators, no philosophy buy-in), and demonstrates the core thesis
— type-directed codegen with zero runtime — in twenty lines of user code.

## Proposal

New package: `@typesugar/tool`.

### Basic usage

```typescript
import { tool } from "@typesugar/tool";

/**
 * Look up current weather for a city.
 * @param city City name, e.g. "Brisbane"
 * @param units Temperature units
 */
@tool
export async function getWeather(
  city: string,
  units: "celsius" | "fahrenheit" = "celsius"
): Promise<WeatherReport> { ... }
```

The macro expands, at compile time, to a static companion:

```typescript
getWeather.tool = /*#__PURE__*/ {
  name: "getWeather",
  description: "Look up current weather for a city.",
  input_schema: {
    type: "object",
    properties: {
      city: { type: "string", description: 'City name, e.g. "Brisbane"' },
      units: { type: "string", enum: ["celsius", "fahrenheit"], default: "celsius" },
    },
    required: ["city"],
  },
  validate: (args: unknown): args is { city: string; units?: "celsius" | "fahrenheit" } =>
    /* inlined structural checks generated by validator<T>() */,
  call: (args: unknown) => { /* validate, destructure, invoke */ },
};
```

No runtime dependency. The schema is a plain object literal; the validator is
inlined structural checks. Everything is derived from the *actual signature* —
it cannot drift.

### Provider adapters

Pure functions (tree-shakeable, no SDK dependency):

```typescript
import { toAnthropicTool, toOpenAITool, toMcpTool } from "@typesugar/tool/adapters";

const tools = [toAnthropicTool(getWeather.tool), toAnthropicTool(searchOrders.tool)];
```

### Collection macro

```typescript
import { toolbox } from "@typesugar/tool";

// Expression macro: collects every @tool-annotated export of the given modules
const tools = toolbox(import("./tools/weather"), import("./tools/orders"));
```

Uses `moduleIndex()` to enumerate annotated functions at compile time and emits a
static array — no dynamic import survives to runtime.

### Refined-type integration

`Refined<number, Positive>` parameters should emit `{ type: "number", exclusiveMinimum: 0 }`,
`Refined<string, Email>` should emit `{ type: "string", format: "email" }`. This rides
on the validate ↔ type-system integration (ROADMAP P4; TODO §Cross-Cutting): the JSON
Schema generator consults `REFINEMENT_PREDICATES` for known brands. Constraints that
have no JSON Schema equivalent fall back to the generated validator only.

## Implementation Plan

- **Wave 1 — schema generation.** `toolSchema<typeof fn>()` expression macro:
  signature → JSON Schema (primitives, literals, unions, arrays, optional params,
  defaults, nested object types). JSDoc → `description` fields. Diagnostics for
  unsupported types (functions, classes, `unknown`) with spans on the offending
  parameter.
- **Wave 2 — `@tool` attribute macro.** Companion emission (`fn.tool = {...}`),
  validator generation via the existing `validator<T>()` path, `call()` wrapper.
- **Wave 3 — adapters + `toolbox()`.** Anthropic / OpenAI / MCP shapes; module
  collection macro.
- **Wave 4 — refined types.** JSON Schema constraint emission from
  `REFINEMENT_PREDICATES` (depends on PEP for validate/refined integration or
  implements the lookup locally as the first consumer).
- **Wave 5 — docs + playground example.** This is the package's marketing surface;
  the playground example should be the README hero.

## Open Questions

1. Should `call()` catch validation failures and return a structured error (model
   retry-friendly) rather than throwing? Recommendation: return
   `{ ok: false, error }` — agents need to feed errors back to the model.
2. Naming: `@typesugar/tool` vs `@typesugar/ai`. Recommendation: `tool` — adapters
   keep it provider-neutral.
3. MCP server scaffold generation (`typesugar mcp-serve ./tools`) — stretch goal,
   separate PEP if Wave 3 lands well.
